ЛЁГКОЕ ТЗ ДЛЯ REPLIT MVP (v0)

0) Цель

Собрать рабочий прототип, чтобы заказчик мог:
	1.	зайти по простому логину
	2.	создать проект через форму ТЗ
	3.	получить ссылку
	4.	видеть ответы исполнителей в одном месте

Исполнитель:
	•	переходит по ссылке
	•	заполняет форму офера
	•	нажимает “Отправить”

⸻

1) Роли

Заказчик
	•	создаёт проекты
	•	получает ссылку
	•	смотрит ответы

Исполнитель
	•	видит задачу
	•	отправляет офер по форме

⸻

2) Поток заказчика

2.1 Мини-регистрация (Login-only)

Экран:
	•	поле: Логин (строка)
	•	кнопка: Войти

Логика:
	•	если логин новый → создать пользователя
	•	если существует → войти
	•	хранить user_id в сессии (cookie)

Без пароля (MVP).

⸻

2.2 Список проектов (простая страница)

Показывает:
	•	список проектов пользователя
	•	кнопка Создать проект
	•	у каждого проекта: название + дата + кол-во оферов

⸻

2.3 Создание проекта = заполнение ТЗ (форма)

Экран “Создать проект”

Поля (минимум)
	1.	Название проекта (обяз.)
	2.	Описание задачи (обяз., многострочное)
	3.	Ожидаемый результат (обяз.)
	4.	Сроки (обяз. текстом или дата)
	5.	Бюджет (необяз.)
	6.	Критерии выбора (необяз. чекбоксы)

Кнопка: Создать проект

Результат:
	•	создаётся проект
	•	статус open
	•	генерируется public_token (для ссылки)

⸻

2.4 Экран проекта (заказчик)

Показывает:
	•	ТЗ проекта (read-only)
	•	публичную ссылку + кнопка Скопировать
	•	список оферов (ответов)
	•	сортировка оферов по дате (минимум)

⸻

2.5 Просмотр оферов (наглядно)

Отображение:
	•	список карточек или таблица

Каждый офер показывает:
	•	контакты исполнителя
	•	подход
	•	сроки
	•	цена
	•	гарантии
	•	риски
	•	дата отправки

(Опционально MVP)
	•	поле “Заметка” заказчика к оферу

⸻

3) Поток исполнителя

3.1 Публичная страница по ссылке /p/{token}

Показывает:
	•	название проекта
	•	ТЗ
	•	кнопку Оставить офер

⸻

3.2 Форма офера (исполнитель)

Поля (обязательные):
	1.	Имя / ник
	2.	Контакт (Telegram/телефон/email)
	3.	Подход (многострочное)
	4.	Сроки
	5.	Цена
	6.	Гарантии
	7.	Риски

Кнопка: Отправить

После отправки:
	•	показать “Офер отправлен”
	•	офер появляется у заказчика в проекте

⸻

4) Данные (минимальная модель)

User
	•	id
	•	login
	•	created_at

Project
	•	id
	•	user_id
	•	title
	•	tz_description
	•	tz_result
	•	deadline
	•	budget
	•	criteria (строка или JSON)
	•	status = open
	•	public_token
	•	created_at

Offer
	•	id
	•	project_id
	•	name
	•	contact
	•	approach
	•	deadline
	•	price
	•	guarantees
	•	risks
	•	created_at

⸻

5) Ограничения MVP (чётко)
	•	без паролей и ролей
	•	без чата
	•	без сравнения в таблице “side-by-side” (можно позже)
	•	без выбора исполнителя (можно позже)
	•	без ИИ
	•	без интеграций HH/Avito

⸻

6) Критерий готовности MVP

MVP готов, если:
	•	заказчик вошёл по логину
	•	создал проект через форму
	•	получил ссылку
	•	исполнитель отправил офер
	•	заказчик увидел офер в проекте

⸻

7) Что тестируем этим MVP
	•	готовы ли заказчики создавать проекты через форму
	•	готовы ли исполнители заполнять офер по форме
	•	достаточно ли “наглядного списка оферов” для выбора
